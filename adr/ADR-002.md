# ADR-002: DOMPurify for XSS Sanitization

- **Date:** 2026-06-23
- **Status:** Accepted
- **Deciders:** Aman Singh

## Context

Seven `dangerouslySetInnerHTML` usages in `BlockRenderer.tsx`, `CourseDetail.tsx`, `BlockEditor.tsx`, and `SalesOnboardingCourse.tsx` rendered HTML strings with zero sanitization.

Four of those strings originated from the `lessons` table in the database. An admin (or anyone who gains admin access) could insert a `<script>` tag via the Course Builder, which would execute in every user's browser when they view that lesson. This is a stored XSS vulnerability with HIGH severity.

## Decision

Install `dompurify` and create a `src/lib/sanitize.ts` wrapper (`safeHtml()`) that sanitizes all HTML strings before they reach `dangerouslySetInnerHTML`. All seven callsites updated.

## Consequences

**Positive:**
- Eliminates stored XSS from database-sourced HTML content
- Single call site to update if sanitization policy changes
- DOMPurify is the industry-standard browser XSS sanitizer; well-maintained, zero-dependency

**Negative:**
- Adds ~22 KB (gzip: 8.8 KB) to the bundle via `purify.es.js` chunk
- DOMPurify strips tags/attributes that it considers unsafe; any admin-authored rich HTML that used unusual attributes may be silently removed (acceptable trade-off)

## Alternatives Considered

- **sanitize-html** — server-side focused; larger bundle; no browser DOM API
- **Manual allowlist** — fragile; XSS attack surface is large and constantly evolving
- **Content Security Policy only** — CSP cannot protect against all stored XSS without `unsafe-inline` restrictions, which would break Tailwind's inline styles
