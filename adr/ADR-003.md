# ADR-003: TanStack React Query for Server State Management

- **Date:** 2026-06-23
- **Status:** Accepted
- **Deciders:** Aman Singh

## Context

All pages used `useEffect` + `useState` pairs to fetch data directly from the Supabase client. Problems:

- Every page remount triggered a new network request with no caching
- Loading and error states were inconsistently implemented (some pages had no error state at all)
- Data was refetched on every render cycle even when it was fresh
- Assessments.tsx ran N serial SQL queries (one per assessment) to fetch question counts — O(N) network round-trips
- No standard pattern for mutations or cache invalidation

## Decision

Adopt `@tanstack/react-query` v5 as the server state layer. All data fetching moves to `useQuery` hooks; all mutations move to `useMutation` hooks. A `QueryClient` is provided at the root via `QueryClientProvider` in `main.tsx`. All Supabase calls are centralized in `src/lib/api.ts`.

## Consequences

**Positive:**
- Automatic caching: repeated visits to a page reuse cached data within the stale window (default 0s stale, 5 min cache)
- Request deduplication: two components querying the same key share one request
- Consistent loading/error state interface across all pages (`isLoading`, `isError`, `error`)
- `queryClient.invalidateQueries()` replaces manual re-fetch patterns
- Supabase client is now confined to `api.ts` — pages have no direct database dependency

**Negative:**
- Adds ~10.5 KB `useQuery` chunk to the bundle (gzip: 3.7 KB)
- Developers unfamiliar with react-query must learn query key conventions and stale/cache lifecycle
- `useMutation` onSuccess/onError callbacks must be careful not to capture stale closure state

## Alternatives Considered

- **SWR** — lighter but less full-featured (no mutations, no query invalidation graph)
- **Redux Toolkit Query** — heavier, requires Redux mental model; over-engineered for this scale
- **Keep useEffect + useState** — rejected; no caching, no deduplication, inconsistent UX
- **Zustand + manual fetch** — adds a client-state store without solving the server-state caching problem

## Query Key Convention

```
['dashboard-data', userId]
['my-courses', userId]
['course-library', userId]
['certificates', userId]
['assessments-data', userId]
['course', courseId]
['lessons', courseId]
['lesson-progress', userId]
```

All user-scoped queries include `userId` as the second key segment so that cache entries are isolated per user (important for shared-device scenarios).
