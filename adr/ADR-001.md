# ADR-001: React Router for Client-Side Routing

- **Date:** 2026-06-23
- **Status:** Accepted
- **Deciders:** Aman Singh

## Context

The original application used a single `useState('dashboard')` to track the current "page" and rendered components via a large if/else switch in `App.tsx`. This pattern:

- Prevented browser back/forward navigation from working
- Made deep-linking impossible (every reload landed on Dashboard)
- Made the URL bar meaningless — users could not bookmark or share views
- Coupled navigation logic into page components via `onNavigate` callbacks

## Decision

Adopt `react-router-dom` v6 for all navigation. Each page maps to a URL route. `<Routes>` and `<Route>` replace the manual switch.

## Consequences

**Positive:**
- Bookmarkable URLs for every page
- Browser back/forward navigation works
- `<NavLink>` in Sidebar highlights active route automatically
- Foundation for future code splitting (route-level `React.lazy`)
- Removes `onNavigate` prop drilling through page components

**Negative:**
- `onNavigate` prop still required by Assessments and CourseLibrary for intra-page sub-navigation; this is tech debt to address in P3
- `AdminPanel` and `CourseBuilder` still receive `onNavigate` for internal section routing

## Alternatives Considered

- **TanStack Router** — more type-safe but significantly more complex to set up; not warranted for this scale
- **Next.js** — file-based routing; would require full framework migration
- **Keep manual state routing** — rejected; no browser integration, no deep-link support
